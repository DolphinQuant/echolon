# WFA-002: WFA pipeline completed only some of its walk-forward windows

## Why this error fires

`WFARunner.run` in `echolon/backtest/wfa/runner.py` raises WFA-002 after the
per-window loop when **some, but not all** windows produced a robust trial. A
window whose `TrialSelector` finds zero surviving trials is skipped (Step 3
`continue`) and never overwrites `selected_robust_trial.json`.

Previously this middle case was silent: the runner kept the windows that *did*
complete, then Step 7's final full-period backtest ran whatever
`selected_robust_trial.json` was last written — i.e. the **last-completed**
window's parameters — over the entire history, and the DRS gates scored on the
shrunken set of completed windows (`G3 = positive / completed`, not
`/ configured`). That silently produces a plausible-but-wrong verdict: a
strategy dead in 3 of 5 windows can still "pass" gates on a 2-window average,
or collapse to DRS 0 because stale params from an earlier window do not
generalise to the full period. Per the No-Misleading-Fallback Policy, an
incomplete WFA must hard-fail rather than emit a misleading score.

The context dict on the exception carries:

- `n_completed` — how many windows produced a robust trial
- `n_total` — how many windows were configured
- `failed_windows` — the window ids that produced no robust trial
- `suggestion` — pointer to per-window `trial_failure_summary.json` artifacts

## Fix

For each window in `failed_windows`, inspect its
`trial_failure_summary.json` (same structured breadcrumb as WFA-001) next to
that window's `optimization_trials.csv`. The usual causes:

- **Survival filter rejects every high-Sharpe trial.** The selector keeps only
  trials with in-sample max-drawdown below the threshold AND a nonzero trade
  count. If a window's best-Sharpe trials all breach the DD cut or stop
  trading, none survive. Widen the parameter range, relax the DD threshold for
  that regime, or address why the strategy can't trade that period.
- **Degenerate "stops-trading" trials dominate.** Zero-trade trials produce no
  drawdown and a clamped Sharpe, so they can crowd the Pareto front; the
  DD-survivor requirement then can't fill the window. An activity floor
  (minimum trades per window) prevents this.

WFA-002 means the strategy could not be robustly evaluated across the full
walk-forward horizon — treat it as a robustness signal, not a transient error.
Re-scoring is only valid once every configured window produces a robust trial.
