# IND-007: Component reads an indicator column not declared in strategy_indicator_list.json

## Why this error fires

`validate_component_smoke` (in
`echolon/strategy/validators/component_smoke.py`) instantiates each
component against a stub engine and calls its trading method once on a
single synthetic bar, recording every name passed to
`self.get_indicator(...)`. If a requested name's **base** does not match
any indicator declared in `strategy_indicator_list.json` (and is not a
system-provided indicator such as `session_phase` / `bars_remaining`), it
is flagged IND-007.

This catches the JSON↔code naming drift that static checks miss: the
engine computes only declared indicators, so a `get_indicator("high_20")`
read when the JSON declared `highest_high` produces an all-NaN /
KeyError column ~15 minutes into the WFA backtest. IND-007 surfaces it in
seconds, before any backtest runs.

This is an **advisory** finding from the standalone `validate_component_smoke`
MCP tool — it is deliberately NOT part of `validate_strategy_full`'s
blocking set.

Finding context:

- `indicator` — the undeclared column the code requested
- `declared` — the indicator bases the JSON actually declared
- `hint` — the two valid fixes

Detection is base-name based (`name == base` or `name.startswith(base + "_")`),
so a legitimate multi-param column like `bbands_upper_timeperiod20_nbdevup2p0`
matches the declared base `bbands` and is NOT flagged. The check errs toward
false negatives over false positives — it only flags a read that shares no
declared base at all.

## Fix

Pick one:

1. **Rename the read** — change the `get_indicator()` argument in the
   component to a column the JSON declares. Confirm the runtime column name
   with `indicator_info(name)` (single-period sweeps emit `name_period`;
   multi-param sweeps emit `name_key1val1_key2val2`).
2. **Declare the indicator** — add it to `strategy_indicator_list.json` so
   the engine computes it. Validate the edited file with
   `validate_indicator_list(payload_json=...)` before re-running.

## Example

Triggering code (entry reads `high_20`, JSON declared `highest_high`):

```python
# strategy_indicator_list.json
# {"highest_high": {"timeperiod": [10, 50]}, "lowest_low": {"timeperiod": [10, 50]}}

class entry_rule(BaseComponent):
    def generate_signal(self):
        breakout = self.get_indicator(f"high_{self.lookback}")  # WRONG base 'high'
        ...
```

`validate_component_smoke` records the read `high_20`, finds no declared
base `high` (declared bases are `highest_high`, `lowest_low`), and flags:

```python
{
  "code": "IND-007",
  "indicator": "high_20",
  "declared": ["highest_high", "lowest_low"],
}
```

Corrected:

```python
        breakout = self.get_indicator(f"highest_high_{self.lookback}")  # matches declaration
```

## Common causes

- Abbreviating a declared name in code (`highest_high` → `high`).
- Renaming an indicator in the JSON without updating the component f-strings.
- Reading an indicator the design intended but forgot to declare.
