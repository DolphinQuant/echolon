# PRM-007: Search range drift between params_to_optimize.json and strategy_params.py

## Why this error fires

A tunable parameter's search range is declared in **two** places that must agree:

- `params_to_optimize.json` — `calculation`/`usage` entries carry a `range`
  (`{"atr_14_period": {"range": [10, 20], ...}}`). This drives the indicator
  calculation + the Optuna search-space construction (and is used at deploy).
- `strategy_params.py` — `optuna_search_space(trial)` calls
  `trial.suggest_int/float("entry_atr_14_period", 10, 20)`. This is what the
  backtest actually optimizes.

PRM-007 fires when the two ranges differ for the same parameter. An EXPLOITATION
value-tune edits `strategy_params.py` directly; if it changes a range there but
not in `params_to_optimize.json` (or vice versa), the indicator-calc searches one
range while the backtest optimizes another — a **silent** inconsistency (no
crash), so it's caught at preflight instead.

Matching is by the **dict key** in `optuna_search_space`
(`entry_params["atr_14_period"]`), which equals the `.json` key — not the optuna
trial name (`"entry_atr_14_period"`), which is inconsistently prefixed.

This is a **validation finding** surfaced by `validate_params_sync` /
`validate_strategy_full`, not a runtime exception. Conservative: non-constant
ranges are skipped (never false-positive).

Finding fields: `param`, `json_range`, `py_range`, `section`.

## How to fix

Make the two ranges identical — edit `strategy_params.py`'s
`optuna_search_space` and `params_to_optimize.json`'s `range` together.
