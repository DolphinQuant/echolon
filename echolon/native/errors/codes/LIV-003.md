# LIV-003: QMT callback returned an unexpected status

## Why this error fires

LIV-003 is the canonical "miniQMT async callback delivered a status the
client cannot map to a normal in-flight transition" code. It is
registered in `echolon.errors.ERROR_CATALOG` and meant to be raised from
the callback path (or from any caller that has just inspected an unusual
broker response) via:

```python
from echolon.errors import raise_error
from echolon.live.config.qmt_constants import QMT_STATUS_MAP

# inside a callback handler
status = getattr(response, "status", -1)
echo_status = QMT_STATUS_MAP.get(status, f"UNKNOWN_{status}")
raise_error(
    "LIV-003",
    seq_id=getattr(response, "seq", 0),
    qmt_status=status,
    echo_status=echo_status,  # always readable, even for unmapped ints
    raw=str(response),
)
```

`UNKNOWN_<N>` for unseen codes ensures an agent parsing the exception
always sees something readable, even when miniQMT ships a new constant.

Context dict:

- `seq_id` — the sequence id miniQMT assigned when the order was submitted
- `qmt_status` — the raw miniQMT integer status
- `echo_status` — echolon-canonical string
  (e.g. `"REJECTED"`, `"CANCELED"`, or `"UNKNOWN_63"` for an unmapped value)
- `raw` — the stringified callback response object

**Wiring status:** the live portfolio runner consumes the QMT order-status
integers directly when processing fills (see
`echolon/live/orchestrator/portfolio.py::_process_fills`, branches on
`status == 56`, `status in (53, 54)`, `status == 57`). The orchestrator
does not currently raise LIV-003 — unmapped codes surface via
`QMT_STATUS_MAP.get(status, f'UNKNOWN_{status}')` in log lines. Future
callback-side handlers that want a typed exception should call
`raise_error("LIV-003", ...)`.

## Fix

First look at `echo_status`:

- `REJECTED` / `CANCELED` / `PARTIAL_CANCELED` — the broker declined the
  order after it was accepted. Pivot to the LIV-002 troubleshooting path.
- `UNKNOWN_<N>` — miniQMT shipped a new status constant the client has not
  been updated to handle. Add an entry to `QMT_STATUS_MAP` in
  `echolon/live/config/qmt_constants.py` and open a follow-up to
  review whether the orchestrator's status branches need to handle it.
- Any other status the client does not expect to see on this callback
  channel — treat as a protocol surprise and file an issue with the full
  `raw` payload.

## Maintenance: extending the status table

If miniQMT introduces a new status integer (say `63 = WAIT_CANCEL`),
extend `QMT_STATUS_MAP` and add a regression test:

```python
# echolon/live/config/qmt_constants.py
QMT_STATUS_MAP: Dict[int, str] = {
    48: "UNREPORTED",
    49: "WAIT_REPORTING",
    50: "SUBMITTED",
    53: "PARTIAL_CANCELED",
    54: "CANCELED",
    55: "PARTIAL_FILLED",
    56: "FILLED",
    57: "REJECTED",
    63: "WAIT_CANCEL",   # new
}
```
