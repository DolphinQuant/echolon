# LIV-003: QMT callback returned an unexpected status

## Why this error fires

`_raise_qmt_callback_error` (in
`echolon/live/platforms/miniqmt/qmt_client.py`) raises LIV-003 when a
miniQMT async callback (order-response, trade-response, etc.) arrives with
a status value the client cannot map to a normal in-flight transition.

The helper deliberately normalises the broker's native integer into an
Echolon-canonical status string (`_QMT_STATUS_TO_ECHO`), and falls back to
`"UNKNOWN_<status>"` for unseen codes ‚Äî so an agent parsing the exception
always sees something readable, even when miniQMT ships a new constant.

Context dict:

- `seq_id` ‚Äî the sequence id miniQMT assigned when the order was submitted
- `qmt_status` ‚Äî the raw miniQMT integer status
- `echo_status` ‚Äî echolon-canonical string
  (e.g. `"REJECTED"`, `"CANCELED"`, or `"UNKNOWN_63"` for an unmapped value)
- `raw` ‚Äî the stringified callback response object

**Note:** `_raise_qmt_callback_error` is defined and tested but the
callback-side wiring is a follow-up ‚Äî as of now LIV-003 is primarily
consumed by unit tests and by callback-level handlers that invoke it
directly after inspecting an unusual response.

## Fix

First look at `echo_status`:

- `REJECTED` / `CANCELED` / `PARTIAL_CANCELED` ‚Äî the broker declined the
  order after it was accepted. Pivot to the LIV-002 troubleshooting path.
- `UNKNOWN_<N>` ‚Äî miniQMT shipped a new status constant the client has not
  been updated to handle. Add an entry to `_QMT_STATUS_TO_ECHO` in
  `qmt_client.py` and open a follow-up to review.
- Any other status the client does not expect to see on this callback
  channel ‚Äî treat as a protocol surprise and file an issue with the full
  `raw` payload.

## Example

Triggering code (a minimal callback handler that surfaces unknown statuses):

```python
from echolon.live.platforms.miniqmt.qmt_client import (
    _raise_qmt_callback_error,
    _QMT_STATUS_TO_ECHO,
)

def on_order_stock_async_response(self, response):
    seq_id = getattr(response, "seq", 0)
    status = getattr(response, "status", -1)
    if status not in _QMT_STATUS_TO_ECHO:
        _raise_qmt_callback_error(seq_id, status, response)
    # otherwise dispatch normally...
```

If a miniQMT update introduces status `63` that isn't in the table, the
callback raises LIV-003 with `echo_status="UNKNOWN_63"` ‚Äî a clear signal
that the translation table needs a new entry rather than a silent mis-dispatch.

Corrected maintenance action (extend the table and add a test):

```python
_QMT_STATUS_TO_ECHO = {
    48: "UNREPORTED",
    49: "WAIT_REPORTING",
    50: "SUBMITTED",
    53: "PARTIAL_CANCELED",
    54: "CANCELED",
    55: "PARTIAL_FILLED",
    56: "FILLED",
    57: "REJECTED",
    63: "WAIT_CANCEL",   # new
}
```
