# DAT-005: Unsupported OHLCV frequency parameter

## Why this error fires

`load_ohlcv` (in `echolon/data/loaders/ohlcv_loader.py`) accepts a
`frequency` keyword argument that resolves the on-disk path layout:

- `"1d"` (default; backward-compat) → `{market_data_dir}/{MARKET}/{asset}/sort_by_date.csv`
- `"1m"` / `"5m"` / `"15m"` / `"1h"` (intraday) → `{market_data_dir}/{MARKET}/{asset}/{frequency}/sort_by_date.csv`

If the caller passes an unrecognized value (e.g. `"1y"`, `"4h"`), the
loader fails early with DAT-005 rather than resolving to a path that
doesn't exist and surfacing a confusing DAT-001 "file not found" — the
caller's intent is clearly invalid, not just missing data.

Per Q48 spec (qorka `docs/3_roadmap/open_questions.md`) + Wave 1A
T23 (echolon commit `e681614`).

Context dict:

- `frequency` — the value the caller passed
- `supported` — the sorted list of accepted frequency strings

## Fix

Pass a frequency value that matches the supported set:

```python
from echolon.data.loaders.ohlcv_loader import load_ohlcv

# Correct:
df = load_ohlcv(market="SHFE", asset="aluminum", market_data_dir=...)  # defaults to "1d"
df = load_ohlcv(market="SHFE", asset="aluminum", market_data_dir=..., frequency="15m")

# WRONG — DAT-005 raised:
df = load_ohlcv(market="SHFE", asset="aluminum", market_data_dir=..., frequency="4h")
```

If a new frequency tier is genuinely needed (e.g., a future paradigm
that operates on 30-minute bars), extend `SUPPORTED_FREQUENCIES`
frozenset + storage convention in `ohlcv_loader.py` rather than working
around DAT-005 at the caller.

## Example

Triggering code:

```python
load_ohlcv(
    market="SHFE",
    asset="aluminum",
    market_data_dir="/path/to/data",
    frequency="1y",  # not in {"1d", "1m", "5m", "15m", "1h"}
)
# DAT-005 raised: frequency='1y', supported=['15m', '1d', '1h', '1m', '5m']
```
