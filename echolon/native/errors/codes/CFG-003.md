# CFG-003: Required path config not injected

## What
A library function (loader, processor, runner) was called without the path
kwarg it needs (`market_data_dir`, `indicator_dir`, `raw_data_dir`, etc.) AND
no equivalent explicit override (`path=`, `metadata_path=`, …) was supplied.

## Why
Echolon library functions no longer fall back to `PathsConfig.from_env()` to
guess the path from `$ECHOLON_PROJECT_ROOT` (or, worse, `cwd`). That fallback
caused silent surprises whenever a caller forgot to inject the right config:
the function would happily read from a different workspace than the caller
intended.

The new contract: **every program constructs one `PathsConfig` at startup and
threads it through.** Library functions never invent paths on their own.

## Fix
Build a `PathsConfig` at the top of your program and pass the relevant field:

```python
from echolon.config.paths_config import PathsConfig

# At program startup, exactly once:
paths = PathsConfig.from_project_root(my_project_root)
# Or, for end-users without a project layout:
# paths = PathsConfig.from_platformdirs("echolon")

# Then thread the specific field into the loader:
df = load_contract_ohlcv(
    market="SHFE", asset="aluminum", contract="al2403",
    market_data_dir=paths.market_data_dir,
)
```

For higher-level orchestrators (`run_indicator_calculation`, `BacktestRunner`)
pass the whole config:

```python
run_indicator_calculation(ctx, output_dir, indicator_list, paths=paths, ...)
```

The CLI entry points (`echolon backtest`, `echolon hello`) build `PathsConfig`
from the workspace marker (`.echolon-workspace.json`) automatically; if you
see CFG-003 from the CLI, the marker is missing or malformed.
