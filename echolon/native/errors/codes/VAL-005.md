# VAL-005: Component method signature doesn't match protocol

## Why this error fires

`validate_component_protocol_signatures` in `echolon/strategy/validators/validate_component_protocol_signatures.py`
raises VAL-005 when it inspects a component class (entry_rule, exit_rule,
risk_manager, or position_sizer) and finds that one of its required methods
has a different signature than the protocol specifies. The mismatch means
the strategy will fail at engine binding time when the framework tries to
invoke the method with the expected arguments.

The context dict on the exception carries:

- `component` — the component name (e.g., "entry_rule")
- `method` — the method that has the wrong signature (e.g., "generate_signal")
- `expected` — the signature the protocol defines
- `actual` — the signature the component actually has

## Fix

Compare the method's signature against the protocol definition and align them.
The protocol specifies exact parameter names and counts — do not add, remove,
or rename parameters.

For example, if the protocol says `generate_signal(self, bar_index: int) -> EntrySignalOutput`,
the component must have exactly that signature. A variant like
`generate_signal(self, bar_index, timestamp)` will trigger VAL-005.

Update the component method to match the protocol exactly, then re-run the
validator.

## Example

Component with mismatched signature:

```python
class EntryRule(BaseComponent):
    def generate_signal(self, bar_index: int, timestamp: datetime) -> EntrySignalOutput:
        # Protocol says (self, bar_index: int), but this has extra 'timestamp' param
        pass
```

Protocol definition:

```python
def generate_signal(self, bar_index: int) -> EntrySignalOutput:
    pass
```

Corrected component:

```python
class EntryRule(BaseComponent):
    def generate_signal(self, bar_index: int) -> EntrySignalOutput:
        # Now matches the protocol exactly
        pass
```
