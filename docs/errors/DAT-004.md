# DAT-004: Trading calendar is empty after generation

## Why this error fires

`CalendarGenerator` (in `echolon/data/transformers/calendar_generator.py`)
builds a per-symbol trading calendar by extracting the `date` column from
raw OHLCV and then filtering by the requested `start_date` / `end_date`
window. If the resulting DataFrame has zero rows, DAT-004 is raised before
the file is written — an empty calendar would later mask itself as a
silent zero-trade backtest.

Three common causes:

1. The source DataFrame has no `date` column (schema drift or wrong file).
2. Every date lies outside the requested range.
3. The source file is literally empty.

Context dict:

- `market`, `instrument` — scope of the generation
- `start_date`, `end_date` — requested window (stringified)
- `rows_seen` — how many input rows were consumed before filtering

## Fix

Inspect the source file and compare its date range with the requested
window. The common error is asking for a window that predates when the
contract started trading.

## Example

Triggering code (asking for a window that predates the contract):

```python
from echolon.data.transformers.calendar_generator import CalendarGenerator

# Aluminum contract data starts in 2015; requesting 2010 yields zero rows
gen = CalendarGenerator(
    market="SHFE",
    instrument="aluminum",
    source_df=aluminum_ohlcv,  # dates span 2015-2025
    output_dir="workspace/data/calendars",
)
gen.generate(start_date="2010-01-01", end_date="2010-12-31")
# DAT-004 raised: rows_seen=0 after filtering
```

Corrected version (window inside the available history):

```python
gen = CalendarGenerator(
    market="SHFE",
    instrument="aluminum",
    source_df=aluminum_ohlcv,
    output_dir="workspace/data/calendars",
)
gen.generate(start_date="2020-01-01", end_date="2024-12-31")  # succeeds
```
