# IND-002: Indicator referenced in code but not declared in JSON

## What
Your strategy code calls `self.get_indicator("ema_50")`, but `ema` is not listed in `strategy_indicator_list.json` (or the period 50 is not declared).

## Why
`strategy_indicator_list.json` is the single source of truth for what gets pre-computed. The indicator pipeline reads the JSON, computes each declared indicator once, and attaches the columns to the bar DataFrame. An indicator not in the JSON is never computed — so `get_indicator` cannot find it at runtime.

This design prevents silently doubled computation and makes the full feature set auditable.

## Fix
Add the indicator to the JSON. Flat-dict format:

```json
{
  "atr": {"timeperiod": 14},
  "rsi": {"timeperiod": 14},
  "ema": {"timeperiod": [20, 50]},
  "market_regime": {}
}
```

Two-element integer lists like `[20, 50]` are inclusive ranges — all periods
from 20 to 50 get computed. Every period you reference in code (`ema_30`, etc.)
must fall inside that range.

To discover valid indicator names + their accepted params, call the MCP tools
`list_indicators(cluster=None)`, `indicator_info(name)`, or use
`echolon.indicators.catalog.info(name)` directly in Python.

## Common Causes
- Adding a new indicator call in code but forgetting to update the JSON
- Using a period in code that isn't in the JSON's period list (e.g., code uses `ema_30` but JSON only has `[20, 50]`)
- LLM agent not aware the JSON exists
